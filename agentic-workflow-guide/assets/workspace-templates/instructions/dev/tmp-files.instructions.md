# Temporary Files Management

調査・検証・デバッグで作る一時ファイルの取り扱いルール。
ワークスペース直下にゴミファイルを溜めないことが目的。

## 1. 配置先の判断フロー

ファイルを作る前に、以下の順で判断する。

```
このファイルは…
├─ 永続的な成果物か？  → 適切な永続フォルダへ（docs/, scripts/, data/ 等）
├─ 再利用可能なツールか？ → scripts/ 配下に正式名で配置
├─ テストとして残すべきか？ → tests/ 配下に正式名で配置
└─ どれでもない（一時的な調査結果・デバッグ出力）
   → tmp/ 配下に作成 (.gitignore 対象)
```

## 2. 命名規約

### ワークスペース直下に作らないファイル名パターン

以下は直下に作成しない。`tmp/` 配下に置くこと。

- `tmp_*.{py,ps1,txt,json,md,pdf,xlsx}`
- `temp_*.{py,ps1,txt,json}`
- `test_*.{py,txt}`（正式なテストは `tests/` 配下に置く）
- `check_*.py`
- `_*.txt` (`_log.txt`, `_loc.txt`, `_origin.txt` などの調査ダンプ)
- `*_output.txt`, `*_result.txt`

### 例外（ワークスペース直下に許可）

- `README.md`, `LICENSE`, `AGENTS.md`, `.gitignore`, `.gitattributes`
- `pyproject.toml`, `package.json`, `docker-compose.yml`
- チームで合意済みの運用ファイル

## 3. セッション終了時のチェック

タスク完了報告の前に、以下を確認する。

1. ワークスペース直下に `tmp_*` `temp_*` `test_*` `check_*` `_*.txt` を作っていないか
2. 作っていた場合、以下のいずれかを実施する
   - 再利用価値あり → `scripts/` `tests/` `docs/` 等へ移動し、適切にリネーム
   - 再利用価値なし → 削除する
   - 判断保留 → `tmp/` へ移動し、最終報告で残件として明示する

## 4. 運用ルール

- 一時ファイルの削除前に、必要なら git 追跡状態と最終更新日を確認する
- tracked な一時ファイルを見つけた場合は、削除だけでなく「なぜ残ったか」を見直す
- 類似スクリプトが複数ある場合は、単発コピーを増やさず既存スクリプトへの統合を優先する
